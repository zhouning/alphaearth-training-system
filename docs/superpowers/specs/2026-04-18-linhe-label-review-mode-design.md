# 临河标注复核模式设计

日期：2026-04-18

## 1. 背景

当前 `Linhe 标注` 已具备第一版能力：
- 浏览未标注 patch
- 单标签标注
- 自动写入 `data/linhe_patches/_labels.parquet`

但在第一轮 1293 张 patch 全量标注完成后，新的问题暴露出来：

> 当前标注台只能处理“未标注 patch”，无法高效回看和修改已标注样本。

而在实际工作中，第二轮更重要的不是继续盲目新增样本，而是：
- 复核“其他”类，把可明确判断的样本重新分流到更具体的类别
- 清洗“道路”“水体”等少数类，提高少样本类别质量
- 必要时对既有标签做覆盖修正

因此，下一步最有价值的升级不是新增复杂功能，而是给现有标注台增加一个 **复核 / 重标模式**。

## 2. 项目定位

本功能定位为一个 **面向已标注 patch 的标签复核与重标工具**。

它不是新的独立产品页，也不是主动学习系统。它的目的很单一：

> 在现有 `Linhe 标注` 页面中，支持用户按模式和类别筛选已标注 patch，并直接覆盖更新旧标签。

## 3. 目标与非目标

### 3.1 目标

1. 在现有 `Linhe 标注` 页面中新增“未标注 / 已标注复核”模式切换
2. 支持按已有标签筛选已标注 patch
3. 支持显示当前 patch 的旧标签
4. 支持点击新标签覆盖保存
5. 保存后自动跳到下一张符合过滤条件的 patch

### 3.2 非目标

1. 不新增第二个独立复核页面
2. 不实现主动学习、难例推荐或模型驱动采样
3. 不支持批量改标签
4. 不支持审阅历史版本或撤销栈
5. 不支持多人审核流

## 4. 推荐方案

推荐采用 **在现有 `Linhe 标注` 页中扩展复核模式** 的方案。

原因：
- 用户已经熟悉当前标注台的交互逻辑
- 可直接复用 patch 预览、标签按钮、保存接口
- 工程代价最小
- 不会把标注工具分裂成两套相似页面

相比之下：
- 新开一个“复核页”会重复大量 UI 逻辑，不值得
- 做主动学习/难例推荐对当前阶段过重，且没有必要

## 5. 交互设计

### 5.1 模式切换
在当前 `Linhe 标注` 页新增一个模式切换控件：

- `未标注模式`
- `已标注复核模式`

行为：
- 未标注模式：延续当前行为，取下一张未标注 patch
- 已标注复核模式：从已标注样本中取下一张符合筛选条件的 patch

### 5.2 类别筛选
当处于“已标注复核模式”时，页面显示一个类别过滤器：

- 全部
- 耕地
- 林地
- 水体
- 建筑
- 道路
- 裸地
- 其他

行为：
- `全部`：从所有已标注 patch 中顺序浏览
- 选具体类别：只浏览当前标签等于该类别的 patch

### 5.3 当前标签展示
在已标注复核模式下，页面要显式显示：
- 当前 patch 路径
- 当前旧标签

这样用户能明确知道当前是在“复核旧标签”，而不是“给新 patch 打标签”。

### 5.4 覆盖保存
无论是未标注模式还是复核模式，点击标签按钮后都调用同一个保存接口：
- 若该 patch 尚未标注，则新增一条标签记录
- 若该 patch 已有标签，则用新标签覆盖旧标签

保存成功后：
- 更新进度统计
- 自动跳到下一张符合当前模式与筛选条件的 patch

## 6. 后端职责

为了支持复核模式，后端标签 API 需要从“只返回未标注样本”扩展到“支持模式与过滤器”。

建议新增或调整参数：

### 6.1 `GET /api/ae/labels/next`
新增可选查询参数：
- `mode=unlabeled|labeled`
- `label=<类别名>`（可选，仅在 labeled 模式下生效）

行为：
- `mode=unlabeled`：返回下一张未标注 patch
- `mode=labeled` 且无 `label`：返回下一张已标注 patch
- `mode=labeled` 且有 `label`：返回下一张旧标签等于该类别的 patch

返回字段建议包含：
- `dataset_id`
- `patch_path`
- `current_label`（未标注模式可为空）

### 6.2 `POST /api/ae/labels/save`
接口无需变更核心语义，只需继续保持：
- 以 `patch_path` 为主键语义
- 再次保存时覆盖旧标签

## 7. 前端职责

前端仅需做最小增强：

1. 增加模式切换控件
2. 增加类别过滤器
3. 调用 `next` 接口时携带 `mode` 和 `label`
4. 在页面中显示 `current_label`
5. 保存成功后继续按当前筛选条件获取下一张

不需要：
- 复杂缓存层
- 批量操作面板
- 分页器
- 拖拽排序或样本队列管理

## 8. 典型使用流程

这轮升级完成后，用户的推荐使用方式是：

1. 进入 `Linhe 标注`
2. 切换到 `已标注复核模式`
3. 先筛 `其他`
   - 把能明确识别的 patch 改标到更具体类别
4. 再筛 `道路`
   - 清洗误标并补强真正道路样本
5. 再筛 `水体`
6. 如有需要，再筛 `林地`

这意味着第二轮工作的重心从“新增标注”转为：
- 标签清洗
- 长尾类别增强
- 结构化复核

## 9. 风险与边界

### 9.1 顺序浏览效率风险
如果复核模式只按文件顺序返回 patch，用户在筛某类时仍可能觉得效率一般。

处理方式：
- 第一版先接受这一点
- 先保证“能筛、能改、能保存”
- 后续再考虑更智能的排序方式

### 9.2 标签覆盖风险
复核模式天然允许覆盖旧标签，因此存在误改风险。

处理方式：
- 第一版不做撤销栈
- 直接覆盖 `_labels.parquet`
- 由用户在操作时确认当前标签与新标签

### 9.3 工程边界
第一版复核模式必须保持简单：
- 只做模式切换
- 只做按标签过滤
- 只做单条覆盖更新
- 不进入复杂审阅产品方向

## 10. 推荐执行顺序

1. 扩展后端 `next` 接口支持 `mode` 和 `label`
2. 扩展前端 `Linhe 标注` 页增加模式切换与类别过滤器
3. 显示当前旧标签
4. 覆盖保存后自动前进到下一张
5. 手工验证“其他”类复核流程

## 11. 最终结论

最合理的下一步，不是继续堆更多新标注，而是把现有 `Linhe 标注` 升级成一个 **复核 / 重标模式**。

这项升级的核心价值是：
- 让第二轮标注真正聚焦在问题类别上
- 帮助减少“其他”类噪声
- 帮助补强道路、水体、林地等长尾类别
- 为下一轮 benchmark 提供更高质量标签

它是当前阶段最小、最务实、最有收益的增强方向。
