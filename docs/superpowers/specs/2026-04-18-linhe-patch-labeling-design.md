# 临河 Patch 地类标注台设计

日期：2026-04-18

## 1. 背景

临河区域版 AlphaEarth 工程闭环已经具备：
- `linhe_patches` 可生成并被平台识别
- 后端训练链路可消费 `linhe_patches`
- 最小 PEFT 训练脚本可运行

当前缺口不在模型链路，而在监督数据。现有 `linhe_finetune.py` 仍使用 pseudo-label 验证流程是否能跑通，这不足以证明该系统对临河真实业务任务有价值。下一步最有价值的动作，是在现有系统里提供一个最小标注能力，让人工能够直接为 patch 生成真实地类标签。

## 2. 项目定位

本功能定位为一个 **为临河区域训练服务的最小 patch 地类标注台**。

它不是一个通用遥感标注平台，也不是一个完整 GIS 绘制系统。它的目的很单一：

> 让用户在 AlphaEarth-System 内直接浏览 `linhe_patches`，并为每个 patch 赋予单一地类标签，形成可直接用于区域模型训练的数据表。

## 3. 目标与非目标

### 3.1 目标

1. 在现有系统里新增一个专用于临河 patch 的标注入口
2. 支持逐张浏览 patch 并进行单标签分类
3. 支持固定类别集的快速点击标注
4. 将标注结果保存为训练可读的结构化标签表
5. 提供最基础的进度统计（已标注数 / 未标注数）

### 3.2 非目标

1. 不支持 polygon / bbox / mask 几何标注
2. 不支持多标签分类
3. 不支持审核流、协作权限、多用户任务分配
4. 不支持自动预标注或主动学习
5. 不尝试在第一版中做通用数据集管理产品

## 4. 类别设计

第一版使用固定的 7 类单标签体系：

1. 耕地
2. 林地
3. 水体
4. 建筑
5. 道路
6. 裸地
7. 其他

选择这套类别的原因：
- 覆盖了当前最重要的农业与自然地表类型（耕地、林地、水体、裸地）
- 保留了建成环境的基础区分（建筑 vs 道路），便于后续分析
- 保留 `其他` 作为兜底类，避免第一版因为类别边界不清而卡住标注效率

这套体系是工程第一版的固定方案，不支持用户自定义类别。

## 5. 交互设计

### 5.1 标注工作流

用户在前端进入一个新的标签页，例如 `Linhe 标注`，完成如下流程：

1. 选择数据集（默认只支持 `linhe_patches`）
2. 系统加载一张 patch 图像
3. 用户观察 patch 内容
4. 点击一个类别按钮完成标注
5. 系统自动保存标签并跳转到下一张未标注 patch
6. 用户可手动选择上一张 / 下一张 / 跳过

### 5.2 界面元素

第一版界面只需要最小元素：
- patch 图像显示区
- 7 个类别按钮
- 当前 patch 路径或编号
- 已标注 / 未标注进度
- 上一张 / 下一张 / 跳过未标注

不需要图层叠加、地图联动、缩放绘制工具、批量筛选器。

## 6. 数据模型

第一版标签存储建议使用结构化表格文件，而不是先引入新的数据库表。建议存储格式为 CSV 或 Parquet，推荐 Parquet。

最小字段：
- `dataset_id`
- `patch_path`
- `label`
- `reviewer`
- `labeled_at`

其中：
- `dataset_id` 第一版固定为 `linhe_patches`
- `patch_path` 与 `_index.parquet` 中保持一致
- `reviewer` 可以先记录当前登录用户或简单用户名
- `labeled_at` 用于后续追踪和增量导出

标签文件建议放在与数据集相邻的位置，例如：
- `data/linhe_patches/_labels.parquet`

这样训练脚本后续可以直接读取 `_index.parquet` 与 `_labels.parquet` 进行 join。

## 7. 前后端职责划分

### 7.1 后端职责

后端只负责最小数据服务：

1. 返回 patch 列表或下一张未标注 patch
2. 读取并返回 patch 图像内容（可转成前端可显示的 PNG/base64）
3. 保存单条标签记录
4. 返回进度统计

后端不负责复杂任务调度，不做自动标注推理。

### 7.2 前端职责

前端负责交互与展示：

1. 加载 patch 图像
2. 展示类别按钮
3. 用户点击后提交标签
4. 保存成功后自动前进到下一张
5. 展示当前进度

前端不承担标签持久化逻辑，只调用后端接口。

## 8. 训练衔接方式

该标注台的价值在于能直接衔接后续训练。

建议后续训练接口按如下方式消费标签：
- 读取 `data/linhe_patches/_index.parquet`
- 读取 `data/linhe_patches/_labels.parquet`
- 用 `patch_path` 关联
- 将有标签 patch 作为监督样本输入分类训练流程

这意味着标注台的输出不是“展示用数据”，而是训练数据资产的一部分。

## 9. 风险与边界

### 9.1 patch 可判读性风险
某些 patch 可能同时包含多种地物，单标签会有模糊样本。

处理方式：
- 第一版接受这种不完美
- 依赖 `其他` 兜底类
- 后续如果发现模糊样本很多，再考虑多标签或细化规则

### 9.2 类别边界风险
林地与灌丛、裸地与施工地、道路与硬化地面之间可能存在边界模糊。

处理方式：
- 第一版先以“主导地物”原则标注
- 暂不引入复杂规则引擎
- 后续再补简单标注说明文档

### 9.3 工程边界
如果一开始尝试做 polygon / mask 标注，复杂度会立即扩大，干扰当前目标。

因此第一版必须坚持：
- 单 patch
- 单标签
- 固定类别
- 自动保存
- 最小进度管理

## 10. 推荐执行顺序

1. 新增后端 patch 标注 API
2. 新增前端 `Linhe 标注` tab
3. 接通 patch 浏览与标签保存
4. 增加 `_labels.parquet` 写入与进度统计
5. 做一轮手工标注演示验证
6. 再决定是否把标签接入监督训练流程

## 11. 最终结论

最合理的下一步不是做复杂几何标注，而是在现有系统中增加一个 **临河 patch 单标签地类标注台**。

第一版类别固定为：
- 耕地
- 林地
- 水体
- 建筑
- 道路
- 裸地
- 其他

这个设计最符合当前工程目标：
- 开发量可控
- 能由你本人直接进行标注
- 结果可以直接服务后续区域模型训练
- 不会把项目拖进通用标注产品的复杂度里
