# 临河真实标签 Benchmark 设计

日期：2026-04-18

## 1. 背景

临河区域版 AlphaEarth 工程闭环已经从“能跑通”进入“能评估价值”的阶段。

当前我们已经完成：
- `linhe_patches` 的构建
- 系统内 patch 标注闭环
- 1293 张 patch 的人工单标签地类标注
- 第一轮基于真实标签的 Houlsby 监督训练

第一轮真实标签训练结果已经说明这条路线具备业务可行性，但仍然缺少一个关键判断：

> 在临河真实标签任务上，Houlsby 相比更轻量的适配方法到底值不值得？

因此，下一步最有价值的工作不是继续扩展功能，而是做一轮小而干净的真实标签 benchmark。

## 2. 项目定位

本工作定位为一个 **面向临河真实标签数据的工程验证型 benchmark**。

它不是论文级大规模实验，也不是完整学术评测框架。它的目的很单一：

> 在统一实验条件下，对比 `linear_probe`、`bitfit`、`houlsby` 三种方法在临河真实标签分类任务上的实际表现，为后续工程路线提供决策依据。

## 3. 目标与非目标

### 3.1 目标

1. 用同一批真实标签 patch 做统一对比实验
2. 比较 `linear_probe`、`bitfit`、`houlsby` 三种方法
3. 输出可解释的结果表格与指标文件
4. 为“是否继续以 Houlsby 为主线”提供证据

### 3.2 非目标

1. 不做多 seed 实验
2. 不引入 `geoadapter` 到本轮 benchmark
3. 不扩展到新的数据集或新的区域
4. 不做论文级统计显著性分析
5. 不在本轮中做任务扩展（如变化检测、分割）

## 4. 方法范围

本轮 benchmark 只包含三种方法：

1. **linear_probe**
   - 只训练头部，作为最弱基线
2. **bitfit**
   - 只调 bias，作为极轻量 PEFT 基线
3. **houlsby**
   - 当前最强候选方法，作为主路线评估对象

不包含 `geoadapter` 的原因：
- 在此前的 Paper12 结果中，它已经表现为负结果路线
- 当前临河真实标签规模仍有限
- 将其纳入只会扩大噪声，不利于当前的工程判断

## 5. 数据与划分策略

### 5.1 数据来源
- 使用 `data/linhe_patches/_index.parquet`
- 使用 `data/linhe_patches/_labels.parquet`
- 仅使用已标注 patch 参与本轮 benchmark

### 5.2 类别范围
第一轮 benchmark 的类别与当前标注体系一致：
- 耕地
- 林地
- 水体
- 建筑
- 道路
- 裸地
- 其他

但如果某些类别在标签文件中没有样本（例如当前林地），则该轮实验只以实际存在的类别为准。

### 5.3 划分原则
为了保证对比可解释，三种方法必须使用：
- 相同的训练/验证划分
- 相同的样本集合
- 相同的标签编码顺序

这意味着划分必须被固定并复用，而不能让每种方法自己重新随机 split。

## 6. 训练控制变量

为了让结果差异主要来自 PEFT 方法，而不是别的因素，本轮 benchmark 必须固定：

- 同一个 Prithvi-100M backbone checkpoint
- 同一个 train/val split
- 同一个 epoch 数
- 同一个 batch size
- 同一个 learning rate
- 同一个损失函数配置（包含是否使用 class weights）

换句话说，只有“方法本身”是变量，其他都应保持不变。

## 7. 输出指标

每个方法至少输出以下指标：

1. `best_val_acc`
2. `weighted_f1`
3. `macro_f1`
4. `per_class_f1`
5. `trainable_params`

这些指标分别回答不同问题：
- `best_val_acc`：整体分类正确率
- `weighted_f1`：考虑类别不平衡后的总体效果
- `macro_f1`：各类平均表现，避免大类掩盖小类
- `per_class_f1`：看具体类别哪些最难学
- `trainable_params`：衡量效果与成本的关系

## 8. 输出产物

建议输出到：

- `results/linhe/benchmark_real_labels.json`
- `results/linhe/benchmark_real_labels.csv`

建议内容包括：
- 方法名
- 各项总体指标
- 参数量
- 各类 F1
- 使用的类名顺序
- 训练配置摘要

如果方便，也可同步输出一个简短 markdown 总结，例如：
- `results/linhe/benchmark_real_labels.md`

## 9. 决策价值

这轮 benchmark 的价值不在于追求“绝对最高精度”，而在于回答以下工程决策问题：

1. Houlsby 是否显著优于 linear probe？
2. BitFit 是否已经足够接近 Houlsby，从而成为更省资源的替代方案？
3. 哪些类别真正受益于更强的 PEFT 方法？
4. 在临河这个真实私有数据场景下，复杂 PEFT 是否值得保留？

## 10. 风险与边界

### 10.1 类别不平衡风险
当前真实标签存在明显类别不平衡，例如道路样本极少。

因此本轮 benchmark 的结果应优先参考：
- `weighted_f1`
- `macro_f1`
- `per_class_f1`

而不能只看总体准确率。

### 10.2 单次随机划分风险
本轮不做多 seed，因此结果仍受单次划分影响。

处理方式：
- 先把它作为工程决策依据
- 如果差距足够明显，再考虑下一轮做 3 seeds

### 10.3 样本规模边界
1293 张 patch 足够做第一轮 benchmark，但还不适合推导非常强的学术结论。

因此该结果应被视为：
- 工程路线验证
- 真实标签条件下的方法筛选
而不是最终论文结论。

## 11. 推荐执行顺序

1. 固定真实标签 train/val split
2. 统一训练脚本支持方法切换
3. 为每种方法运行一轮训练
4. 提取并汇总指标
5. 输出 JSON / CSV 结果表
6. 写一段简短结论，回答 Houlsby 是否值得继续保留

## 12. 最终结论

最合理的下一步，不是继续无约束地做更多训练，而是先做一个 **临河真实标签的三方法 benchmark**：

- `linear_probe`
- `bitfit`
- `houlsby`

这一轮 benchmark 的核心价值，是帮助当前项目做工程决策，而不是追求论文级完整性。只要这轮实验能产出一张清晰可解释的对比表，就足以支撑下一阶段的路线判断。
