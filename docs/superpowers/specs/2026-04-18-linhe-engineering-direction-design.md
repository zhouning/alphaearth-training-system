# 临河区域版 AlphaEarth 增强项目设计

日期：2026-04-18

## 1. 背景

Paper12 已经基本回答了当前项目在方法层面的核心问题：在 Prithvi-100M 上，不同 PEFT 方法对异构输入的适配能力存在明显差异，其中 Houlsby adapter 效果最稳定，LoRA 基本失效，GeoAdapter 作为独立输入级适配方法呈现负结果。

与此同时，新增的临河样例数据提供了一个新的机会：不是继续追求新的 foundation model 方法创新，而是验证并落地一条更贴近实际业务的问题链路——如何把小范围、私有、高分辨率遥感数据接入 AlphaEarth-System，形成一个面向特定区域的增强能力。

## 2. 项目定位

本项目定位为一个 **区域版 AlphaEarth 工程落地项目**。

它不是一个“重新训练基座模型”的研究项目，也不是一个以新 PEFT 方法为核心卖点的论文项目。它的核心目标是：

> 将临河私有遥感影像转化为 AlphaEarth-System 中可训练、可推理、可展示、可迭代优化的区域增强能力。

这里的“区域增强能力”指：
- 能把临河数据转换为统一 patch 数据集
- 能在现有 Prithvi-100M / AlphaEarth 技术栈上完成区域微调
- 能在平台中作为一个实际可用的数据集和训练入口出现
- 能支持后续区域任务，如地类识别、变化检测、农业场景监测等

## 3. 目标与非目标

### 3.1 目标

1. 接入临河样例数据，形成统一场景目录与 footprint 清单
2. 生成适合训练的 patch 数据集，作为平台中的区域数据源
3. 基于现有 Prithvi-100M + Houlsby PEFT 路线，完成最小可跑的区域训练链路
4. 在 AlphaEarth-System 中展示“私有区域数据 -> 区域模型增强”的工程闭环
5. 在实现过程中保留足够的实验记录，为未来可能的应用论文留出空间

### 3.2 非目标

1. 不尝试从零训练新的 foundation model
2. 不把当前阶段定义为新的方法论文开发
3. 不为了论文而额外引入高风险、大范围的新算法改造
4. 不在当前阶段承诺区域模型具有跨区域泛化能力

## 4. 核心技术路线

### 4.1 数据路线

临河样例数据的主要特征是：
- 空间范围集中于临河区域
- 影像分辨率较高
- 主体为 3 波段 RGB 成品影像
- 数据适合作为私有区域数据，而不适合作为 foundation model 预训练语料

因此，数据路线应分为两层：

**第一层：区域训练数据层**
- 扫描并整理所有场景元数据、足迹、多季度分布
- 统一切片为可训练 patch
- 建立 `linhe_patches` 这样的标准化数据集入口

**第二层：区域增强数据层**
- 在后续阶段尝试与 Sentinel-2 / AlphaEarth embedding 对齐
- 将临河高分 RGB 数据作为私有增强层叠加到通用底座之上

### 4.2 模型路线

当前阶段采用保守且已被 Paper12 支持的方案：

- Backbone：Prithvi-100M
- 主 PEFT 方法：Houlsby Adapter
- 输入适配：GeoAdapter 或 zero-pad 作为 RGB -> 6 通道桥接手段
- 输出目标：优先服务区域任务，而不是构造新的通用表征模型

之所以选择这条路线，是因为：
- Houlsby 是当前代码库中已有实验支持的最佳方案
- GeoAdapter 已不再作为主方法候选，而是退回到“输入桥接工具”角色
- 这条路线与工程落地目标一致，风险最低，复用现有成果最多

### 4.3 平台路线

工程闭环应体现在平台中，而不是只停留在离线脚本：

1. 临河数据能被平台识别为一个数据集
2. 前端可选择该数据集进行训练
3. 后端训练流程能消费该数据集
4. 训练输出可以沉淀为区域模型或中间结果
5. 后续可扩展到展示区域效果、比较基线与增强结果

## 5. 论文策略

当前默认策略是：**不以论文为主目标**。

Paper12 已经承担了方法层面的主要论文价值。本项目后续的工作不应再被强行包装成新的方法创新研究，否则会带来不必要的实验压力和方向偏移。

但这不意味着完全放弃论文化可能性。合理的策略是：

- 工程落地优先
- 保留实验记录、指标、可视化、失败案例
- 只有在后续出现明确亮点时，再考虑将成果整理为应用型或案例型论文

潜在可写方向包括：
1. 全球遥感基础模型在区域私有高分数据上的适配案例研究
2. 私有高分 RGB 数据作为 AlphaEarth 增强层的工程与实验评估
3. 面向区域场景的遥感大模型本地化部署与微调系统

这些都属于 **应用研究 / 系统应用 / 区域适配案例**，而不是强方法创新。

## 6. 成功标准

本项目阶段性成功不依赖论文录用，而依赖以下工程结果：

1. 临河数据可稳定扫描、整理并形成标准目录
2. 可生成训练 patch，并形成平台可识别的数据集
3. 区域 PEFT 训练链路可运行
4. 平台中可以选择临河数据并触发训练流程
5. 能清楚回答以下问题：

> AlphaEarth-System 是否能够利用私有区域数据，形成一个服务于小范围场景的 AlphaEarth 增强能力？

如果这个答案是“可以，而且流程稳定、可复用、结果可解释”，那么项目目标已经成立。

## 7. 风险与边界

### 7.1 数据边界
- 当前数据主要是 RGB，光谱维度有限
- 因此不应夸大其对 foundation model 的表征贡献
- 更合适的定位是“高分辨率区域细节增强”而不是“重建通用遥感表征”

### 7.2 研究边界
- 当前阶段不再围绕“提出一个新方法”展开
- 如果没有显著实验亮点，不强写论文
- 论文只是副产品，不反向决定工程节奏

### 7.3 工程边界
- 优先做最小闭环，而不是一次性做完整产品
- 每一步都应服务于“区域增强能力可跑通”这个核心目标
- 避免过早引入大规模自动标注、复杂多模态耦合或过重的系统改造

## 8. 推荐执行顺序

1. 完成临河数据清单与 patch 构建闭环
2. 打通平台中的数据集识别与训练入口
3. 跑通基于 Houlsby 的最小区域 PEFT 训练
4. 补充可视化与效果验证
5. 再决定是否需要做 Sentinel-2 / AlphaEarth embedding 对齐
6. 最后根据实际增益决定是否形成应用型论文

## 9. 最终结论

临河样例数据结合 Paper12 后续工作的价值，**主要体现在工程落地与区域适配能力构建上**，而不是新的方法学创新。

因此，后续工作最合理的方向是：

- 把它作为一个高价值工程项目推进
- 在过程中保留论文可能性
- 但不让论文诉求主导项目节奏

这会让项目更稳，也更符合当前数据条件、已有实验结论和实际业务目标。
